# MiniArena：多人实时战斗服务器项目需求说明书

## 1. 项目概述

### 1.1 项目名称

MiniArena——基于 C++ 的多人实时战斗服务器。

### 1.2 项目定位

本项目是一个面向游戏服务器岗位的个人工程项目，模拟多人实时竞技游戏的核心服务器链路，包括：

- 客户端长连接管理；
- 玩家登录与会话管理；
- 匹配和房间管理；
- Tick 驱动的战斗逻辑；
- 玩家操作同步；
- 战斗状态广播；
- 断线检测与重连；
- 网络异常模拟；
- 自动化压测和性能优化。

项目不以完成复杂玩法或客户端画面为主要目标，而是重点验证游戏服务器在以下场景中的表现：

- 大量玩家同时在线；
- 玩家高频发送操作指令；
- 多房间并发运行；
- 单个热点房间压力过大；
- 延迟、丢包和抖动等异常网络环境；
- 客户端断线和集中重连；
- 慢客户端造成发送队列积压。

### 1.3 项目背景

实时多人游戏服务器需要同时处理大量长连接、玩家输入、战斗计算和状态广播。与普通请求—响应式 Web 服务相比，游戏服务器不仅关注平均吞吐量，还关注 Tick 是否稳定、尾延迟是否可控，以及不同玩家之间的状态是否一致。

结合协议栈实习中积累的网络通信、连接管理、数据传输和性能分析经验，本项目将这些底层能力应用于实时游戏业务，重点研究以下问题：

1. 如何使用事件驱动网络模型承载大量客户端长连接；
2. 如何将网络收发与游戏逻辑解耦；
3. 如何保证一个房间内的战斗逻辑有序执行；
4. 如何控制多人状态同步产生的广播开销；
5. 如何在异常网络和高并发场景下保持服务器稳定；
6. 如何通过压测数据定位瓶颈，并验证优化效果。

### 1.4 项目价值

该项目需要体现的不是“做出了一个小游戏”，而是以下工程能力：

- Linux 网络编程能力；
- 高并发服务器设计能力；
- 游戏服务器业务建模能力；
- 多线程和并发控制能力；
- 协议设计能力；
- 性能测试能力；
- 故障定位能力；
- 数据驱动的性能优化能力。

## 2. 项目目标

### 2.1 核心目标

实现一套可运行、可测试、可观测、可扩展的多人实时战斗服务器。

系统应支持：

- 玩家注册或模拟登录；
- TCP 长连接；
- 心跳检测；
- 玩家匹配；
- 创建和加入战斗房间；
- 多个战斗房间并发运行；
- 玩家移动、普通攻击和技能释放；
- 固定频率 Tick；
- 玩家输入处理；
- 战斗状态广播；
- 断线检测；
- 一定时间内断线重连；
- 战斗结算；
- 自动化机器人压测；
- 性能指标采集；
- 网络异常测试。

### 2.2 非目标

第一版不实现以下内容：

- 完整图形客户端；
- 复杂物理引擎；
- 大世界无缝地图；
- 商城和付费系统；
- 公会、聊天、邮件等外围系统；
- Kubernetes 集群部署；
- 完整分布式事务；
- 生产级账号安全系统；
- 复杂反作弊系统；
- 完整帧同步回滚算法。

这些内容会增加开发量，但不一定能明显提高游戏服务器面试中的项目价值。

### 2.3 预期性能目标

| 指标 | 第一阶段目标 | 优化阶段目标 |
|---|---:|---:|
| 稳定在线连接数 | 2,000 | 10,000 |

## 3. 用户与业务场景

### 3.1 玩家角色

玩家通过测试客户端或机器人客户端连接服务器，执行以下流程：

1. 建立连接；
2. 完成登录；
3. 进入在线状态；
4. 发起匹配；
5. 被分配到战斗房间；
6. 加载完成；
7. 进入战斗；
8. 周期性发送移动、攻击或技能指令；
9. 接收其他玩家和房间状态；
10. 完成战斗；
11. 接收结算结果；
12. 返回大厅或退出。

### 3.2 管理与测试角色

测试程序需要能够：

- 创建指定数量的机器人；
- 控制连接建立速率；
- 控制机器人操作频率；
- 控制每个房间人数；
- 模拟随机掉线；
- 模拟集中重连；
- 统计请求延迟；
- 统计消息成功率；
- 输出测试报告。

### 3.3 典型业务场景

#### 场景 A：普通多人战斗

10 名玩家进入一个房间，每名玩家每秒发送 5 次移动或攻击指令，服务器以 20 Hz 更新战斗状态。

#### 场景 B：多房间并发

5,000 名玩家分布在 500 个房间中，各房间独立执行战斗逻辑。

#### 场景 C：热点房间

大量玩家集中在同一个房间或同一区域，用于观察广播开销、Tick 抖动和单线程负载上限。

#### 场景 D：重连风暴

大量玩家在短时间内断开连接并同时重连，用于测试网关、登录状态恢复和连接资源释放。

#### 场景 E：慢客户端

部分客户端故意降低读取速度，观察服务端发送队列是否持续积压，以及是否影响其他正常客户端。

#### 场景 F：异常网络

模拟延迟、丢包、重复包、乱序和抖动，验证心跳、序列号、重连和消息容错机制。

## 4. 系统总体架构

### 4.1 第一版架构

```text
┌──────────────────────┐
│ Test Client / Robot  │
└──────────┬───────────┘
           │ TCP
           ▼
┌──────────────────────┐
│    Gateway Server    │
│ 连接、心跳、编解码、转发 │
└──────────┬───────────┘
           │ 内部消息
           ▼
┌──────────────────────┐
│     Game Server      │
│ 登录、匹配、房间、战斗   │
└───────┬────────┬─────┘
        │        │
        ▼        ▼
┌────────────┐ ┌────────────┐
│   Redis    │ │   MySQL    │
│ 会话与缓存  │ │ 持久化数据  │
└────────────┘ └────────────┘
```

### 4.2 为什么不一开始拆成大量微服务

个人项目一开始拆成登录服、匹配服、场景服、战斗服、聊天服和数据库代理，会产生大量服务治理和通信代码，但核心战斗逻辑可能反而完成度不足。

因此采用渐进式设计：

第一阶段：

- Gateway Server；
- Game Server；
- Robot Client；
- Redis；
- MySQL。

第二阶段可拆分为：

- Lobby Server；
- Match Server；
- Battle Server。

### 4.3 数据流

```text
客户端生成 PlayerMoveCmd
        ↓
添加消息头和序列号
        ↓
TCP 发送至 Gateway
        ↓
Gateway 完成拆包、校验和会话识别
        ↓
指令投递到 Game Server
        ↓
根据 room_id 定位 Room
        ↓
放入 Room 输入队列
        ↓
下一个 Tick 统一处理
        ↓
更新玩家位置
        ↓
生成状态变化
        ↓
广播给房间内相关玩家
```

## 5. 技术选型

### 5.1 开发语言：C++20

选择原因：

1. 游戏服务器和高性能网络服务中应用广泛；
2. 可直接使用 Linux Socket、epoll 和线程接口；
3. 能体现内存管理、对象生命周期和并发控制能力；
4. 便于使用 ASan、TSan、perf 和 gdb；
5. 与协议栈实习背景衔接自然。

机器人压测客户端可使用 Go：

- C++：核心服务器；
- Go：压测机器人和测试控制器。

### 5.2 操作系统：Linux

使用：

- Socket；
- epoll；
- eventfd；
- timerfd；
- `std::thread`；
- TCP/UDP；
- perf；
- gdb；
- tc/netem。

### 5.3 网络 I/O：epoll + Reactor

采用多 Reactor 模型：

```text
              ┌──────────────┐
连接建立 ────▶│ Main Reactor │
              └──────┬───────┘
                     │ 分配连接
       ┌─────────────┼─────────────┐
       ▼             ▼             ▼
┌────────────┐ ┌────────────┐ ┌────────────┐
│IO Reactor 1│ │IO Reactor 2│ │IO Reactor N│
└────────────┘ └────────────┘ └────────────┘
```

Main Reactor 负责：

- 监听端口；
- 接受连接；
- 将连接分配给 I/O 线程。

I/O Reactor 负责：

- 读取网络数据；
- 粘包和半包处理；
- 协议解析；
- 消息校验；
- 发送队列；
- 连接关闭；
- 心跳状态更新。

### 5.4 网络协议：第一版 TCP，第二版扩展 UDP

第一版使用 TCP，先完成完整业务闭环。

第二版可将高频、可覆盖的状态同步切换到 UDP，例如：

- 玩家位置；
- 朝向；
- 动画状态；
- 非关键环境状态。

可靠性要求高的消息仍走 TCP。

### 5.5 序列化协议：Protocol Buffers

```proto
syntax = "proto3";

message MessageHeader {
```

## 6. 核心模块需求

### 6.1 网络模块

功能要求：

- 非阻塞 Socket；
- epoll；
- 连接生命周期管理；
- 粘包和半包处理；
- 收发缓冲区；
- 心跳检测；
- 连接超时；
- 消息大小限制；
- 异常连接隔离。

消息帧格式：

```text
+----------------+----------------+----------------+
| total_length   | message_id     | flags          |
| 4 bytes        | 4 bytes        | 2 bytes        |
+----------------+----------------+----------------+
| sequence                        | payload        |
| 8 bytes                         | N bytes        |
+---------------------------------+----------------+
```

安全限制：

- 单包最大 64 KB；
- 发送队列设置上限；
- 单连接消息频率限制；
- 非法消息超过阈值后断开；
- 未登录连接不得调用战斗接口。

### 6.2 会话模块

状态：

```text
CONNECTED
   ↓
AUTHENTICATED
   ↓
IN_LOBBY
   ↓
MATCHING
   ↓
IN_ROOM
   ↓
IN_BATTLE
   ↓
SETTLEMENT
```

负责：

- 连接与玩家身份绑定；
- session_id 生成；
- 重复登录处理；
- 心跳记录；
- 断线状态保存；
- 重连校验；
- 会话过期清理。

### 6.3 匹配模块

第一版采用简单匹配：

- 玩家选择模式；
- 进入等待队列；
- 人数达到要求时创建房间；
- 分配 Room ID；
- 返回房间信息。

### 6.4 房间模块

```cpp
class Room {
public:
    RoomId room_id;
    RoomState state;
    std::unordered_map<PlayerId, Player> players;
    CommandQueue command_queue;
    uint64_t current_tick;
    TimePoint next_tick_time;
};
```

状态：

```text
CREATED
   ↓
WAITING_PLAYERS
   ↓
LOADING
   ↓
RUNNING
   ↓
SETTLING
   ↓
DESTROYED
```

### 6.5 Tick 模块

```text
Tick Rate = 20 Hz
Tick Interval = 50 ms
```

流程：

```text
Tick 开始
   ↓
提取当前待处理指令
   ↓
按照玩家和序列号校验
   ↓
执行移动、攻击和技能逻辑
   ↓
更新 Buff、冷却和死亡状态
   ↓
计算需要同步的状态变化
   ↓
生成广播消息
   ↓
记录 Tick 耗时
   ↓
Tick 结束
```

### 6.6 战斗模块

玩家属性：

- 玩家 ID；
- 位置；
- 朝向；
- 当前生命值；
- 最大生命值；
- 攻击力；
- 防御力；
- 移动速度；
- 技能冷却；
- 存活状态。

操作类型：

- 移动；
- 停止移动；
- 普通攻击；
- 释放技能；
- 离开战斗。

### 6.7 AOI 模块

采用九宫格或固定网格，仅同步当前网格及相邻网格中的玩家状态。

### 6.8 广播模块

关键消息：

- 进入战斗；
- 技能结果；
- 玩家死亡；
- 战斗结束；
- 结算结果。

可覆盖消息：

- 玩家位置；
- 朝向；
- 动画；
- 非关键环境状态。

### 6.9 重连模块

流程：

1. 连接关闭；
2. 玩家进入断线状态；
3. 房间保留角色 30 秒；
4. 新连接携带 session token；
5. 验证身份；
6. 恢复绑定；
7. 下发完整战斗快照；
8. 继续增量同步。

## 7. 并发模型设计

### 7.1 线程划分

```text
1 个 Accept 线程
N 个 I/O 线程
M 个 Logic 线程
1 个 DB 异步线程池
1 个指标线程
1 个日志线程
```

### 7.2 跨线程通信

第一版使用：

```cpp
std::mutex + std::condition_variable + std::queue
```

确认成为瓶颈后再考虑：

- 无锁 MPSC 队列；
- 批量取消息；
- eventfd 唤醒；
- 每线程本地队列；
- 对象池。

### 7.3 对象生命周期

- Connection 由所属 I/O 线程管理；
- Room 由所属 Logic Worker 管理；
- 跨线程传递稳定 ID；
- 避免跨线程裸指针；
- 关闭连接使用明确状态机。

## 15. 项目中的重点取舍

### 15.1 先正确，再优化

```text
单线程正确
   ↓
多房间正确
   ↓
多线程正确
   ↓
压力测试
   ↓
定位瓶颈
   ↓
针对性优化
```

### 15.2 单房间串行，多房间并行

一个房间固定归属一个 Logic Worker，多个房间可并行运行。

### 15.3 Redis 和 MySQL 不参与每个 Tick

战斗中以内存计算为主，结束后异步持久化。

### 15.4 性能数据必须可复现

## 17. 简历项目描述模板

**MiniArena 多人实时战斗服务器**

- 基于 C++20 和 Linux 实现多人实时战斗服务器，使用 epoll 和多 Reactor 模型完成长连接管理、心跳检测、协议编解码及异步发送，客户端与服务端通过 Protobuf 通信。
- 设计 Gateway 与 Game Server 分层架构，采用固定 Logic Worker 管理战斗房间，通过消息队列解耦网络 I/O 与游戏逻辑，保证单房间内玩家操作按序执行。
- 实现匹配、房间状态机、20 Hz Tick、移动攻击、技能计算、状态广播、AOI、断线重连及战斗结算，使用 Redis 保存会话与在线状态，使用 MySQL 持久化战绩。
- 开发参数化机器人压测工具，模拟登录、匹配、操作、掉线及集中重连，围绕连接数、P95/P99 延迟、Tick 耗时、CPU、内存和网络吞吐建立测试方案。
- 使用 tc/netem、perf、AddressSanitizer 和 ThreadSanitizer 模拟异常网络并定位性能与并发问题，通过房间串行化、批量广播、AOI 和发送队列限流完成优化，并以重复压测验证效果。
